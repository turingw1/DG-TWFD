# A100 Migration Checklist

This note fixes the code revisions and runtime prerequisites required to move
the active DG-TWFD work back to an A100 server.

## Scope

This checklist is for:

- the root DG-TWFD repository
- the CTM submodules tracked by the root repository
- the separate EDM experiment repository under `refs/edm`

It is not a general environment spec for every historical branch.

## Important Repository Fact

`refs/ctm` and `refs/ctm-cifar10` are tracked as git submodules by the root
repository.

`refs/edm` is **not** tracked as a submodule by the root repository. It is a
separate git repository that must be cloned or updated independently on the
server. Cloning only the root repository is not enough for the EDM-side
experiments added during this round.

## Step 3: Prepare Python Environments

There are two practical layers here.

### Root project

The root project has only a minimal dependency declaration in
`pyproject.toml`. It is enough for code import and part of the local training
stack, but not enough to fully define the EDM experiment runtime.

Recommended baseline:

```bash
cd /data2/yl7622/Zhengwei/DG-TWFD
conda activate consistency
python -m pip install -e '.[dev]'
python -m pip install torchvision numpy scipy matplotlib pillow torch-fidelity
python -m pip install -e '.[teacher]'
```

### EDM experiments

Run EDM-side experiments from `refs/edm` and make the repo importable:

```bash
cd /data2/yl7622/Zhengwei/DG-TWFD/refs/edm
conda activate dg
export PYTHONPATH="$PWD:${PYTHONPATH:-}"
python -m pip install click tqdm pillow scipy requests psutil matplotlib
```

If the server image does not already contain CUDA-enabled PyTorch, install the
correct torch wheel for that server before running EDM.

## Step 4: Prepare Cache And Public Assets

The following runtime assets are **not** stored in git:

- EDM pretrained checkpoints
- FID reference `.npz` files
- Inception detector cache used by `fid.py`
- generated sample folders
- generated figures and result tables
- datasets

Recommended cache setup:

```bash
export DNNLIB_CACHE_DIR=/data2/yl7622/Zhengwei/DG-TWFD/.cache/dnnlib
mkdir -p "$DNNLIB_CACHE_DIR"
```

For the current EDM experiment READMEs, the first online run will fetch public
artifacts automatically if the server can access the internet. If the server
cannot, copy the files in advance and replace the config values with absolute
local paths.

## What Git Currently Covers Well

The current git state is sufficient for:

- root training and evaluation code
- root active documentation under `docs/experiments/DG_TWFD_v3/`
- fixed CTM submodule revisions
- EDM experiment source code for:
  - `experiments/dg_twfd_teacher_proxy/`
  - `experiments/dg_twfd_timewarp_analysis/`
  - including the qualitative suite, baseline sweep, and multi-trajectory time-warp heatmap analysis

## What Git Does Not Cover

The current git state does **not** fully capture:

- conda environment lockfiles
- server-local CUDA and driver assumptions
- downloaded checkpoints and public references
- DNNLIB cache contents
- dataset mirrors and server-local dataset paths
- generated experiment outputs
- unpublished local files such as diagnostic screenshots

## Minimum Manual Notes To Keep With Each Server Run

For every major run, record these fields in the experiment log or run folder:

- root repo commit
- `refs/edm` commit
- active conda env name
- `torch` version
- CUDA version
- exact command
- checkpoint path
- FID ref path
- dataset path
- output root

Without these fields, later comparison across servers becomes fragile even if
the source code is committed.

## Recommended Next Hardening Step

If server migration is going to be frequent, add one of the following next:

1. a root-level `environment_a100.yml` for the main project and an
   `refs/edm/environment_edm.yml` for the EDM side, or
2. a single bootstrap shell script that:
   - checks out the fixed revisions
   - exports cache variables
   - verifies imports
   - prints the exact smoke commands

At the moment, the biggest structural gap is not missing code commits. It is
that `refs/edm` is maintained as a separate repository and the runtime
environment is documented but not lockfile-driven.
