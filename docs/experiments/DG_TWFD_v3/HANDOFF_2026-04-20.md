# DGTD v3 Handoff

This handoff is for continuing work on the `DG_TWFD_v3` branch without replaying
the full conversation history.

## 1. Collaboration Workflow

The current workflow is evidence-driven:

1. Codex reads local code, configs, and experiment docs.
2. Codex makes focused code or documentation changes locally.
3. Codex runs only lightweight local checks, such as `py_compile` and targeted
   `pytest`.
4. Codex commits each completed change on `DG_TWFD_v3`.
5. The project owner runs training, sampling, and evaluation on the server.
6. The project owner returns logs, metrics, tracebacks, sample summaries, and
   file listings.
7. Codex analyzes the returned evidence, updates code/docs, and commits the next
   focused patch.

Server experiments are not run locally by Codex. Long-running validation belongs
to the server loop.

## 5. Current Algorithm State

The intended mainline is:

- online teacher data path
- online teacher trajectory anchors
- affine / Jacobian-lite online continuation as the DGTD residual main source
- symmetric half-stopgrad residual
- defect-guided time warp
- cached continuation only as fallback, not the primary path

The important current code path is:

1. `DGTDTrainer` loads real images when online teacher data is enabled.
2. `TeacherAdapter.online_trajectory_from_x0(...)` builds an online teacher
   trajectory for the batch.
3. `TeacherAdapter.local_flow(...)` uses online trajectory anchors to return
   source `online`.
4. `compute_dgtd_residual(...)` builds the symmetric residual.
5. `scripts/run_sample_dgtd.py` samples from random noise using the learned map
   and learned warp.

## 6. How `src/` Reached The Current Version

The current `src/` tree is not a clean independent rewrite. It was built by
retaining the useful `dgfm` infrastructure and replacing the training objective
and teacher path step by step.

High-level lineage:

1. The starting point was the existing `dgfm` / map-branch infrastructure:
   - config loading
   - dataset builders
   - distributed launcher support
   - map-model construction
   - sampling / evaluation entrypoints
   - checkpoint and JSONL logging conventions
2. `src/dgtd/` was introduced as the DGTD-specific layer:
   - `cache.py` for trajectory payloads and interpolation
   - `warp.py` for monotone defect-guided time density
   - `metrics.py` for unified sigma-aware weights and detail metric
   - `defect.py` for the DGTD residual and density target
   - `teacher.py` for online / cached continuation routing
   - `train_dgtd.py` for the DGTD trainer
3. Early DGTD v3 used offline/frozen teacher trajectory cache semantics.
4. Round 2 fixed the residual from a one-sided detach path to symmetric
   half-stopgrad, so both direct and bridge branches receive training signal.
5. Round 3 audited why online teacher data was available but online continuation
   was not the dominant residual source.
6. Round 4 promoted online trajectory anchors into the continuation mainline:
   - source `online`
   - affine / Jacobian-lite continuation
   - cached continuation only as fallback
7. The latest fix corrected online teacher trajectory semantics:
   - dataloader images are clean images
   - they must be forward-noised before becoming the `u=0` teacher state
   - previous `e402a` trained on the wrong trajectory direction and is invalid

When continuing, do not assume every old `src/` path is equally relevant. The
active path is the `dgtd_map` trainer and DGTD sampler/evaluator wiring.

## 8. Most Recent Critical Finding

The first full run `e402a` trained for 50 epochs but produced noise-like
samples. Its logs looked numerically healthy, but the run is considered invalid.

The root cause was teacher trajectory semantics:

- old behavior:
  - dataloader clean images were passed directly as the first teacher trajectory
    state
  - `u=0` became identical to the clean image
  - the student learned a self-consistent but sampling-invalid trajectory
- evidence returned by the project owner:
  - `mse_input_u0 = 0.0`
  - samples had very low neighbor correlation
  - training loss and defect decreased but sampling remained noise-like

The fix is committed:

- `ac506d8 Fix online teacher clean trajectory semantics`

After the fix:

- clean images are mapped from `[0,1]` to the teacher domain
- the teacher scheduler forward-noises them to construct the `u=0` noisy state
- DDIM rollout then builds the trajectory toward `u=1`
- returned endpoint check showed:
  - `u0` is noise-like
  - `mse_input_u0` is no longer zero
  - `u1` is image-domain-like

Old `e402a` results should not be used to tune losses.

## 9. Current Utility Script

Use `scripts/analyze_dgtd_run.py` to compress long run outputs into actionable
diagnostics.

Typical command:

```bash
python scripts/analyze_dgtd_run.py \
  --run-root "$RUN_ROOT" \
  --eval-root "$METRIC_ROOT" \
  --sample-root "$SAMPLE_ROOT" \
  --config "$FM_CONFIG" \
  --output "$RUN_ROOT/dgtd_analysis.md" \
  --json-output "$RUN_ROOT/dgtd_analysis.json"
```

This script summarizes:

- train / val trends
- online continuation source usage
- warp diagnostics
- teacher trajectory cost
- sample tensor statistics
- checkpoint key structure
- likely next blockers
