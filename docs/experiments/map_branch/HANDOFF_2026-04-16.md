### 3.2 Earlier long-run quality evidence

The project repeatedly showed the following pattern:
- losses could go down steadily
- `train/val` curves could look stable
- but samples still looked like:
  - blurry structure
  - colored speckles
  - no stable convergence to clean images

One representative earlier finding:
- a 29-epoch run reached approximately:
  - `train_loss: 0.1168 -> 0.0118`
  - `val_loss: 0.2906 -> 0.1420`
  - `val_pixel_loss: 0.1655 -> 0.0718`
  - `val_perceptual_loss: 0.2604 -> 0.1611`
  - `val_endpoint_loss: 0.4001 -> 0.1996`
- but long-step FID still stayed in the `330+ / 340+ / 370+` style range,
  depending on checkpoint and sample count

Interpretation:
- consistency-style optimization was real
- generation quality still lagged badly
- therefore low loss should never be treated as sufficient evidence of success

## 7. Key technical details that will affect future work

These points are easy to miss and materially affect debugging and research.

### 7.1 `teacher_sampler` does not use dataset images as semantic supervision

This was already surfaced in diagnostics:
- `target_uses_dataset_images=false`

Meaning:
- the current online teacher target builder mainly uses the batch size / class
  label structure, not the actual CIFAR image content, when generating teacher
  targets
- therefore train/val loss should never be read as ordinary image-conditioned
  generalization loss

This is one reason low loss can coexist with bad FID.

### 7.2 Timewarp is auxiliary, not primary

Current timewarp:
- learns a monotone time reparameterization
- is trained with a separate defect-driven objective
- changes where training/sample time points lie

But earlier evidence strongly suggests:
- timewarp is not the primary root-cause fixer
- it cannot rescue bad main target semantics by itself

### 7.3 Endpoint loss is also not primary

Earlier `tw` experiments suggest:
- endpoint can change behavior slightly
- but it is not the dominant explanation for poor sample quality

### 7.4 Target build is the runtime bottleneck

This has consistently shown up in logs:
- `train_target_build_sec` dominates `train_forward_sec`

So future speed work should prioritize:
- teacher target construction efficiency
- teacher rollout batching / sub-batching
- avoiding unnecessary repeated target-build work

More gains are likely to come from target-build optimization than from generic
network micro-optimizations.

### 7.5 Dual-GPU training currently keeps eval single-GPU on purpose

This is intentional:
- train on `0,1`
- leave `2,3` free
- eval/sample/export/metrics on one GPU only

Do not “optimize” this by casually spreading eval onto all GPUs; it would break
the intended mixed-use server policy.

### 7.6 The latest throughput tuning is aggressive

Current defaults are intentionally aggressive:
- `batch_size=160`
- `teacher_sampler_sub_batch=128`
- for `e603a`:
  - `endpoint_batch_size=64`
  - `timewarp_batch_size=32`

These values were chosen because server feedback said memory was still underused
at `20-25G` per A6000. They are not yet fully validated.

Future node should treat them as:
- a throughput trial
- not yet a settled optimum

### 7.7 Activation checkpoint is intentionally retained

The project explicitly did **not** solve DDP by deleting expensive modules or
disabling checkpointing wholesale.

Current intended status:
- activation checkpoint stays enabled
- DDP compatibility is handled by:
  - non-reentrant checkpoint
  - avoiding second backward-like DDP paths through the wrapped model

If future nodes face DDP trouble again, the first response should be:
- inspect DDP graph usage carefully
- not immediately drop modules
